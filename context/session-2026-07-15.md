## 1. Phase 6 — vector store becomes a cache, then deploy ✅

### The serverless statelessness problem

The in-memory vector store (`lib/store.ts`) does not survive Vercel: `/api/index`
and `/api/ask` can land on different serverless instances, so a visitor would index
a document and then get *"that document is no longer indexed"* on their question.

**Fix — store is a cache, not the source of truth.** The client already has the
document text on screen, so it sends it with each question. A cache hit skips the
work; a **miss re-embeds inline and answers anyway** (~$0.00002/question). The
`documentId` guarantee is unchanged: a miss rebuilds **only from the text that
caller supplied**, never from another visitor's cached copy.

- `lib/indexer.ts` (new) — `indexDocument(text, replacing?)`, shared by `/api/index`
  and the `/api/ask` rebuild-on-miss path, so the two cannot chunk the same text
  differently.
- `/api/ask` — on a store miss, rebuilds from the supplied `text`; with no text to
  rebuild from, it fails **honestly** with a 409 rather than guessing.
- `app/page.tsx` — sends `{ question, documentId, text }` with each ask.

Verified with a simulated cold instance: fake `documentId` + text → correct answer;
without text → honest 409. Warm cache still hits (embed = a few query tokens only,
no wasteful rebuild).

### Deploy to Vercel

- Imported `enn05/docuquery-app`; Next.js auto-detected, default build settings.
- **Both env vars must be set in Vercel** (Production + Preview + Development):
  `ANTHROPIC_API_KEY` and `VOYAGE_API_KEY`. The build succeeds without them, then
  every API call 500s — this is the step that silently breaks a demo.
- **Gotcha hit live:** `{"error":"Server is missing a valid embeddings API key."}`
  meant the Voyage key was missing/misapplied. **Env var changes require a redeploy**
  — Vercel injects them at build/boot, not into a running deployment.

## 2. Scanned / image-only PDFs — a deliberate limitation (not a bug)

A converted-image PDF (a scan, no text layer) returns a specific 422:
*"No readable text found. This looks like a scanned PDF…"* — see
`app/api/extract/route.ts`. This is correct behaviour: handing the model an empty
string would invite a hallucinated summary. The failure is made **visible**, matching
the app's whole thesis.

Documented it in the README "Known limitations" section. Reading scans is a solved
problem — **Claude is multimodal**, so the pages could be sent as images — gated
behind a per-page cost cap. Sketched as a possible **Phase 7** (see
`context/` discussion / plan notes): the likely simplification is Claude's native
PDF `document` content block, which may remove the need for a rendering library
entirely. Not built.

Also added the app screenshot to the README (`docs/screenshot.png`).

## 3. Stretch goal — streaming the summary token-by-token ✅

Chosen stretch goal: **streaming responses** (of the three: tool calling,
multi-document, streaming).

### Scope decision (deliberate): summarize-only

Streamed `/api/summarize` **only**, left `/api/ask` untouched. Reason: `/api/ask`
strips citations pointing at excerpts the model was never given, validating the
*whole* answer before sending. Streaming tokens before that validation runs would
risk flashing a citation the user can't open — violating the "never render
unvalidated model output" invariant. Summaries have no post-processing and are the
longest output (~124 out-tokens vs ~33 for an answer), so that's where streaming is
felt most.

### How it works

**Server (`app/api/summarize/route.ts`)**
- `messages.create` → `messages.stream()`, piping `text_delta` events into a
  `ReadableStream` returned as `text/plain`.
- **First-event pre-flight.** Returning a streaming `Response` commits the status to
  200, so auth/rate-limit/400 failures are surfaced on the first `iterator.next()`
  and mapped through the existing `errorResponse()` **before** 200 commits. Status
  mapping unchanged.
- **Mid-stream cost logging.** Cumulative usage is tracked from `message_start`
  (input tokens) and `message_delta` (running output tokens), so an abort after
  tokens were billed still records the spend.

**Client (`app/page.tsx`, `handleSummarize`)**
- Reads `res.body` and appends chunks live.
- Added a `textRef` stale-guard (this handler previously lacked one) so editing text
  mid-stream cancels the reader instead of resurrecting stale output.
- On a mid-stream error, clears a truncated summary rather than showing it as final;
  final `TextDecoder` flush; explicit `res.body` guard.

### Verification
- `npm run lint` + `npm run build` clean; all 5 routes intact.
- Live stream confirmed via a timestamped Python client: summary arrives in **4
  chunks over ~2.7s**, not one blob.
- Cost log fires: `[cost] /api/summarize in=… out=…`.
  (**Note:** in Next.js 16 dev, route `console.info` goes to
  `.next/dev/logs/next-development.log`, not the piped `npm run dev` stdout.)
- Error paths return JSON with correct codes: empty→400, over-limit→413, bad JSON→400.

### Code review (via `/code-reviewer`)
Two Medium findings, both fixed before commit:
1. **Cost-audit hole** — a mid-stream failure billed tokens but logged nothing (the
   exact gap `lib/api.ts` was hardened against). Fixed by logging partial usage.
2. **Partial-as-final** — a truncated summary was left on screen next to a vague
   error. Fixed by clearing it on the error path.
Plus two Lows (decoder flush, `res.body!` assertion) folded in.

Merged as PR #3.
